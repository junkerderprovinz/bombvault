Fixes off-site restore-drill false failures and makes a failed drill say why and which check.

## Fixed

- **The off-site disaster-recovery drill no longer fails a perfectly restorable backup.** (#30 / #37, thanks @manilx and @BaukeZwart) The drill restores your newest off-site snapshot into a sandbox and cross-checks it against restic's own accounting. It was requiring restic's `stats --mode restore-size` file count to equal restic's `ls` file count, but those two restic counters legitimately differ on real data (hardlinks, and how restore-size tallies files), so a restore that reproduced the exact file count and bytes was still flagged as a "verification mismatch" (e.g. `restic reports 988 files / N bytes, sandbox has 988 files / N bytes`). The drill now verifies what actually matters: the on-disk restored file count equals restic's `ls` count, and the on-disk bytes match restic's restore-size to within a few KB. A truncated or partial restore is still caught.

## Improved

- **A failed restore drill now tells you why and which check.** The dashboard's red "restore drill" row and off-site pill now show the actual reason and label it as the off-site DR restore (distinct from the green local integrity check), and the Settings drill line shows the recorded reason for a past run too. Previously a failure was a reason-less red.
- **Run the off-site DR check from the dashboard.** The failing off-site DR row now has a "Run off-site DR check" button, so a manual run of the same check the dashboard reflects can clear the red once it passes (a manual local integrity check runs a different check and can't clear the off-site row).
- **A scheduled drill that is skipped because the repo is busy now records a dated reason** instead of silently leaving the old state frozen, so the dashboard no longer shows an unexplained stale red after a skipped run.

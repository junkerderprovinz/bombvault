Two bug fixes: backups no longer start stopped dependency containers, and config backups can now be deleted.

## Fixed

- **A backup no longer starts a "stop during backup" dependency that was already stopped.** (#33, thanks @ptmorris1) The container you back up already followed detect-state (a stopped one stays stopped), but the *other* containers you configure to stop for the duration of a backup (e.g. a database an app depends on) were being restarted unconditionally afterward. Because stopping an already-stopped container is a no-op success at the Docker daemon, an already-off dependency landed in the "restart these" list and got started. In a backup-all of an app plus its stopped database it could even cascade (the database gets started, then its own backup sees it running and restarts it too). Dependencies now follow the same detect-state rule as the main container: only the ones that were actually running before the backup are stopped and restarted.
- **Config backups can now be deleted.** (#34, thanks @ptmorris1) The Config page listed its self-backup snapshots but had no way to remove one — the delete engine already supported the config domain, it just wasn't exposed. Each config snapshot now has a delete button, matching the flash and container pages. Off-site append-only repositories still refuse deletes (the refusal is surfaced as an error, as elsewhere).

Two big additions: a first-class **files backup domain** so BombVault backs up arbitrary host folders, and **restore from another BombVault repo**, so you can move a container, VM or file set from one server to another. Bundles issues #61, #62 and #63.

## Added
- **Files backup domain (#62).** A fourth first-class domain alongside containers, VMs and the flash drive, with its own **Files** tab. Add named **file sets** (a label plus a host folder such as a cache or array share), each with its own restic exclude patterns. Full parity with the other domains: schedules, retention, off-site replication, integrity and disaster-recovery drills, dashboard protection card and heatmap, notifications and the Storage card. Restore a file set in place after an explicit confirmation, or into a folder you choose. Sources must be visible under the container's `/mnt` mapping, which the Unraid template's default Host Data mount already covers.
- **Restore from another BombVault repo (#61).** A new card on the Recovery page opens another BombVault instance's backup repository, using that instance's `APP_KEY`, as a one-time read-only session. Browse its containers, VMs and file sets and restore any of them onto this server. The session lives only in memory and expires by itself, nothing is ever written to the other repository, and your own backup settings stay untouched. Mount the other server's backup share and point the card at it. (Live server-to-server federation is out of scope.)
- **Off-site verification on the dashboard (#63).** The off-site integrity drill, the only off-site proof VMs can carry, now shows its own "off-site verified" badge on each domain's protection card, next to the local-verify and disaster-recovery badges.

## Changed
- The heatmap, per-domain Healthchecks and the Storage card all cover the new files domain.
- Default repository layout is now `/mnt/user/bombvault/{container,vms,flash,config,files}`.

## Fixed
- **Failed container backups are now visible (#64).** When a whole-batch fault (a lost repo mount, a full disk, a restic error) hit some containers, those failures were not recorded, so the dashboard showed nothing and the notification was a bare "N failed". Every container failure is now recorded with its reason, shows up in the run history and heatmap, and the scheduled notification names which containers failed and why.

## Notes
- A file set cannot be renamed once it has backups (its snapshots are tagged by name), so pick the name up front or create a new set.
- Existing installs migrate automatically; the new domain and the foreign-restore card are opt-in and off until you enable them.

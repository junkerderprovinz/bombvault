Two fixes: the What's-new dialog loads again, and a late-mounting backup share no longer looks like your backups vanished.

## Fixed

- **The "What's new" dialog shows the changelog again.** It fetched the notes from GitHub at runtime, which BombVault's own Content-Security-Policy (`connect-src 'self'`) blocks — so it always fell back to an error. BombVault now serves its own release notes locally (embedded in the app), so the dialog works offline and without GitHub rate limits. The "View on GitHub" link is still there. (#54)
- **A backup share that mounts late at boot no longer looks like your backups are gone.** If BombVault started before a remote backup share (e.g. under `/mnt/remotes`) was mounted, the list showed no local backups and a new backup could fail — or, worse, write an empty repo over the mount point. BombVault now recognises an established repo whose backing store has vanished, refuses to re-initialise over it (no empty-repo shadowing), and shows a clear "backup path not mounted yet" message instead of "no backups". (#55)

## Changed

- **The Docker template mounts `/mnt` as rslave (Read/Write - Slave).** So a remote share that mounts *after* the container starts becomes visible to BombVault without a restart. Existing installs: re-apply the template from Community Apps, or set the **Host Data** mount's Access Mode to **Read/Write - Slave**, to get the no-restart recovery. (#55)
